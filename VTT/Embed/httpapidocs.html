Basic usage concepts:

Submit a GET request to receive this page.
Submit a POST request to issue API interactions.
For WebSockets, send plaintext containing the same contents as the POST request body. Alternatively, may send JSON as plaintext with the same KV pairs. To include the AUTH token, append it as the auth parameter. Whenever a server processes the WS message, it will respond with a JSON object (plaintext), that, at the very least, will include the statuscode of the result under the status key. Successful method results will also include their method identifier under the method key.

Authorization:

A GET request to receive this page does not require authorization.
To use the API anonimously, submit POST requests without a Authorization header. Most API request require an authorized user. Those that do not, will specify a
Anonimous
tag. If an API request is made that requires an authorized user, but no Authorization was provided, the API will return a 401 - Unauthorized status code. If an API request is made that requires an authorized user, and the authorization provided was invalid, the API will return a 401 - Unauthorized status code. If an API request is made that requires an authorized user, and the credentials do not have premission for the operation, the API will return a 403 - Forbidden status code.
To authorize a client, include the Authorization header, where the header content is either Basic BASE64ENCID:SECRET, or Auth AUTHTOKEN. The BASE64ENCID:SECRET is a Base-64 encoded string, where the non-encoded content is ID:SECRET. ID is the GUID of the client to authorize as, and SECRET is base64 encoded secret. If authorization was correct, the server will respond 200 OK with content that is the AUTH token for the current session. A session lasts as long as the VTT server is online. When the session terminates, the AUTH token provided is no longer valid. If authorization was not correct, the server will send a 401 - Unauthorized status code.

KeepAlive and WebSocket:

If KeepAlive was specified, the server will return a 426 - Upgrade Requried status code, indicating that the client should upgrade the connection to a WebSocket. To upgrade to a WebSocket, send any HTTP request with a Upgrade: websocket Connection: Upgrade headers. The server will respond with a 101 - Switching Protocols response, with the headers being the same.

HTTPS:

Secure connections and WSS are currently not supported.

API Methods:

Authorize

Anonimous

POST: method=authorize&ID=%GUID%&secret=%B64SECRET%
WS: { "method": "Authorize", "id": "%GUID%", "secret": "%B64SECRET%" }

Sends an authorization request to the server.

Parameters:
method: must be Authorize.
id: The GUID of the client, standard GUID format.
secret: The base-64 encoded secret of the client.

Returns:
200 - OK if authorization successful, with the body being the AUTH key to use for the current session.
401 - Unauthorized if invalid.
WS: { "status": STATUSCODE, "auth": "AUTHKEY" }

Message

Authorized
Action
Noreturn

POST: method=message&message=%MSG%
WS: { "method": "message", "auth": "%AUTH%", "message": "%MSG%" }

Sends a chat message to the server from a specified user. The message syntax is exactly the same as if it were sent by the actual client. The client does not need to be online to send this message. Note that the MSG parameter may not include the & symbol if sent by POST or non-json WS. It may be escaped with a \a syntax.

Parameters:
method: must be message.
message: The chat message to send.

Returns:
401 - Unauthorized if no authorization was provided.
200 - OK if the operation was successful.
WS: { "status": STATUSCODE }
